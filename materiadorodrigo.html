REST(Representational State Transfer ou tranferência de estado representacional) é um conjunto de regras e diretrizes arquiteturais usado na programação para que diferentes sistemas conversem entre si pela internet. Em vez de ser uma tecnologia, é um modelo de boas práticas que utiliza o protocolo padrão da web(HTTP). REST; Regras/restrições REST Cliente servidor(client-server) Sem estado(stateless) Cacheable Rest e HTTP HTTP é o protocolo de transporte de dados usado na internet, enquanto resto é um estilo arquitetural de regras para criar serviços web, geralmente ultilizando o HTTP, HTTP faz a conexão, e Rest define o padrão de uso(como GET, POST, DELETE) para transferencia de dados. Rest é como organizar e HTTP é como transportar. O que são APIs? (Application Programming Interface, ou interface de programação de aplicações) É um conjunto de regras, protocolos e ferramentas que permite que diferentes softwares e sistemas se comuniquem entre si. Como funciona? Ela funciona recebendo uma requisição (pedido) de um sistema (cliente), processando-a acessando um banco de daos ou servidor, e retornando uma resposta. Onde usamos api? -Login com google -Aplicativos bancários -Mapas e geolocalização -Previsão do tempo -Redes sociais Tipos de API: -Públicas -Privadas -Parceiras -Internas Toda API descente tem um WAF tomando conta. WAF serve para aplicação web e API. WAF(Web Application Firewall) é uma solução de segurança que protege aplicações web(sites/APIs) filtrando e monitorando o tráfego HTTP/HTTPS, agindo como um proxy reverso entre a internet e o servidor. Ele bloqueia ameaças comuns, como injeção de SQL, Cross-Site Scripting(XSS) e bots, diferenciando-se de firewalls de rede por focar na camada de aplicação. Os tokens são assinados usando um segredo privado ou uma chave pública/privada. O que são métodos ágeis Conjunto de práticas para desenvolvimento rápido e interativo. Valoriza: -Colaboração com o cliente -Resposta rápida a mudanças -Software funcionando mais do que documentação extensa -Exemplos: Scrum,XP,kanban. Scrum Como funciona na prática: Product Owner define o backlog: "Cadastrar usuário","validar senha","Enviar e-mail de boas-vindas"Sprint Planning: a equipe escolhe o que vai fazer em 2 semanas.Ex: nesse Sprint,farão o cadastro e calidação de senha. Daily Scrum: Reuniões diárias de 15 min para sincronizar o progresso. Sprint review: Apresentação da funcionalidade funcionando ao final da sprint. Sprint Retrospective: Equipe analisa o que funcionou e o que pode melhorar. Resumo: Ciclos curtos e planejados (sprints), com entregas frequentes e colaboração constante. XP (Extremme Progamming) Como funciona na prática: -TDD(test-driven Development): Antes de implementar o cadastro, escrevem um teste:"Usuário deve ter e-mail válido". Depois escrevem o código para passar no teste. -Pair Programing(Programação em par): Dois desenvolvedores sentam juntos, um escreve código, o outro revisa em tempo real. -Refatoração constante: Melhoram o código sempre que possível, sem mudar o comportamento. -Integração contínua: Código novo é integrado e testado automaticamente várias vezes ao dia. -Feedback frequente com o cliente: Mostram pequenas partes funcionando a cada poucos dias. Kanvan Como criar um quadro Kanban com colunas: -Criam um quadro kanban com colunas: A fazer -> Em progresso -> Concluído -As tarefas são representadas por cartões(ex:"Corrigir bug no cadastro","Melhorar layout"). -Limitam o número de tarefas por colina : max 2 tarefas "em progresso". -A equipe puxa as tarefas conforme termina as anteriores (modelo puxado). -O foco é no fluxo continuo em vez de ciclos fixos. Resumo: Visual, flexível, ideal para suporte e manutenção contínua de software. Práticas ágeis com OO -Design Simples: foco no necessário, OO ajuda na organização -Responsabilidade única (SRP): cada classe -Código Limpo: nomes claros, baixa acoplamento -Padrões de Projeto(design patterns) auxiliam na manutenção Exemplo prático -Aplicação simples de cadastro de usuários O princípio F.I.R.S.T.
